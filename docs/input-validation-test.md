# 入力バリデーション強化 動作検証記録

## 検証日
2026-07-16

## 対象
Issue #31 / PR #33（入力バリデーション強化）

## 検証環境
`docker compose up --build`（worktree: `PlannerwithExpense-validation`、フレッシュな新規ボリューム）

## 検証方法
ユーザー登録→取得したアクセストークンを使い、`curl`でバックエンドAPI（`/api/expenses`、`/api/schedules`、`/api/budgets`）に対して正常系・異常系のリクエストを送信し、期待どおりのステータスコード・エラーメッセージが返るかを確認した。

## 検証内容と結果

### 支出（`/api/expenses`）

| 項目 | 期待結果 | 実際の結果 |
|------|---------|-----------|
| 正常な支出登録（金額1000円、カテゴリ「食費」） | 201 | OK（201） |
| マイナス金額（-500） | 422で拒否 | OK（`amount: Input should be greater than 0`） |
| ゼロ金額（0） | 422で拒否 | OK（`amount: Input should be greater than 0`） |
| 上限超過の金額（999,999,999円） | 422で拒否 | OK（`amount: Input should be less than or equal to 100000000`） |
| 空白のみのカテゴリ | 422で拒否 | OK（`category: Value error, カテゴリを入力してください`） |
| 256文字のカテゴリ（上限255文字） | 422で拒否 | OK（`category: String should have at most 255 characters`） |
| 1001文字のメモ（上限1000文字） | 422で拒否 | OK（`memo: String should have at most 1000 characters`） |

### 予定（`/api/schedules`）

| 項目 | 期待結果 | 実際の結果 |
|------|---------|-----------|
| 正常な予定登録 | 201 | OK（201） |
| 終了日時が開始日時より前 | 422で拒否 | OK（`Value error, 終了日時は開始日時以降にしてください`） |
| 空白のみのタイトル | 422で拒否 | OK（`title: Value error, タイトルを入力してください`） |
| 繰り返し終了日が開始日より前 | 422で拒否 | OK（`Value error, 繰り返し終了日は開始日以降にしてください`） |

### 予算（`/api/budgets`）

| 項目 | 期待結果 | 実際の結果 |
|------|---------|-----------|
| 正常な予算登録（PUT） | 200 | OK（200） |
| 不正な月（13） | 422で拒否 | OK（`month: Input should be less than or equal to 12`） |
| マイナスの貯蓄目標 | 422で拒否 | OK（`savings_target: Input should be greater than or equal to 0`） |

### エラーレスポンス形式の統一

すべての異常系リクエストで、レスポンスの`detail`が単一の文字列で返ることを確認した（FastAPIの`RequestValidationError`用に追加した例外ハンドラが機能している）。これにより、既存の`HTTPException`（文字列の`detail`）とフロントエンド側で同じロジックで表示できる。

## 補足
- `npm run build`がエラーなく成功することは別途確認済み。
- 検証時、メインリポジトリ側の`docker compose`スタックは一時停止し、検証後に復旧している（ポート競合回避のため）。
- ブラウザ実機でのエラー表示確認は、当初ブラウザ自動操作ツールがなく未実施だったが、Playwrightを用いて追加検証した（下記「ブラウザ実機確認（追加検証）」を参照）。

## ブラウザ実機確認（追加検証）

### 検証日
2026-07-16

### 対象
Issue #42（上記API検証で未実施だったブラウザ実機でのエラー表示確認）

### 検証環境
`docker compose up -d`（worktree: `PlannerwithExpense-validation`）+ Playwright（Chromium）によるブラウザ自動操作

### 検証方法
Playwrightで実際のブラウザ（Chromium）を起動し、ユーザー登録→カレンダー画面から各モーダル（支出・予定・予算）を開いてフォームに異常な値を入力し、送信時に赤字のエラーメッセージ（`.error-message`）が画面上に表示されることをスクリーンショット付きで確認した。

ブラウザのHTML5標準バリデーション（`min`/`max`/`maxlength`属性によるネイティブの入力チェック）は、ユーザーが実際に画面を操作する場面では一次防御として機能し、フォームの`submit`イベント自体をブロックする。今回の検証ではアプリ側のバリデーションロジック（フロントエンドJS・バックエンドAPI）が正しく動作し、エラーが画面に表示されることを確認する目的で、`form.noValidate = true`をテストスクリプトから設定してHTML5標準バリデーションを意図的に迂回した上で送信している。

### 検証内容と結果

| 項目 | 検証方法 | 表示されたエラーメッセージ |
|------|---------|---------------------------|
| 支出: マイナス金額（-500） | フォーム入力→送信（フロントエンドJS検証） | 「金額は1円以上100,000,000円以下で入力してください」 |
| 支出: 空白のみのカテゴリ（自由入力） | フォーム入力→送信（フロントエンドJS検証） | 「カテゴリを入力してください」 |
| 支出: 256文字のカテゴリ | 通信を改ざんしてフロントエンド検証を迂回し、バックエンドの422エラーをそのまま表示させる | 「category: String should have at most 255 characters」 |
| 予定: 空白のみのタイトル | フォーム入力→送信（フロントエンドJS検証） | 「タイトルを入力してください」 |
| 予定: 繰り返し終了日が開始日より前 | フォーム入力→送信（フロントエンドJS検証） | 「繰り返し終了日は開始日以降にしてください」 |
| 予算: マイナスの貯蓄目標 | フォーム入力→送信（フロントエンドJS検証） | 「貯蓄目標は0円以上100,000,000円以下で入力してください」 |

いずれのケースも、フォーム下部に赤字（`.error-message`）でエラーメッセージが表示され、モーダルは閉じずに再入力可能な状態を維持することを確認した。

「支出: 256文字のカテゴリ」のケースでは、Playwrightのネットワークインターセプト機能でリクエストボディを送信直前に改ざんし、フロントエンドの文字数チェックを迂回した状態でバックエンドに送信した。この場合もバックエンドが422エラーを返し、そのエラーメッセージ（`RequestValidationError`ハンドラが整形した文字列）がそのままモーダルに表示されることを確認した。これにより、フロントエンドの入力チェックが何らかの理由で迂回された場合でも、バックエンド側のバリデーションとエラー表示の仕組みが最終防御として機能することを確認できた。

### 補足
- Playwright・Chromiumは本リポジトリには追加せず、検証用の一時環境（リポジトリ外のディレクトリ）にインストールして使用した。
- 正常系（正しい値での登録が成功しモーダルが閉じること）についても、ユーザー登録後のカレンダー画面遷移などを通じて併せて確認している。
